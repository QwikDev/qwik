<objective>
Build the variable migration analysis module that decides whether module-level declarations should be moved into segments, re-exported as _auto_VARNAME, or kept at root.

Purpose: Module-level variables referenced by segments use a different mechanism than scope-level captures. This module implements the decision tree: move (single-use, safe), re-export (shared/exported/side-effects), or keep (not used by segments). This feeds into Plan 03 where the decisions are applied to codegen.

Output: `src/optimizer/variable-migration.ts` with `analyzeMigration()` function and comprehensive tests.

   export interface MigrationDecision {
     action: 'move' | 'reexport' | 'keep';
     varName: string;
     targetSegment?: string;  // symbolName of the segment (only for 'move')
     reason: string;
   }

2. Export `ModuleLevelDecl` interface:
   ```typescript
   export interface ModuleLevelDecl {
     name: string;
     /** Start position of the full declaration statement in source */
     declStart: number;
     /** End position of the full declaration statement in source */
     declEnd: number;
     /** The declaration text (e.g., `const helperFn = (msg) => console.log(msg)`) */
     declText: string;
     /** Whether the declaration has `export` keyword */
     isExported: boolean;
     /** Whether the initializer may have side effects */
     hasSideEffects: boolean;
     /** Whether this name comes from a destructuring pattern with other bindings */
     isPartOfSharedDestructuring: boolean;
     /** The kind of declaration: var, let, const, function, class */
     kind: string;
   }
   ```

3. Export `collectModuleLevelDecls()` function:
   ```typescript
   export function collectModuleLevelDecls(
     program: any,
     source: string,
   ): ModuleLevelDecl[]
   ```
   Walk program.body top-level statements to collect VariableDeclaration, FunctionDeclaration, and ClassDeclaration nodes. For each:
   - Extract all binding names (handle destructuring via recursive pattern walk)
   - Determine `isExported` from ExportNamedDeclaration wrapper or `export` keyword
   - Determine `hasSideEffects` conservatively: an initializer is SAFE (no side effects) ONLY if it is:
     - A Literal (string, number, boolean, null, undefined, bigint, regexp)
     - An ArrowFunctionExpression or FunctionExpression
     - A TemplateLiteral with only literal quasis and no expressions
     - An ObjectExpression where ALL property values are safe
     - An ArrayExpression where ALL elements are safe
     - An Identifier referencing another known-safe variable (conservative: skip this, just mark as side-effect)
     - Everything else (CallExpression, NewExpression, MemberExpression, etc.) → hasSideEffects = true
   - Determine `isPartOfSharedDestructuring`: true if the VariableDeclarator uses ObjectPattern or ArrayPattern AND that pattern has more than one binding

4. Export `analyzeMigration()` function:
   ```typescript
   export function analyzeMigration(
     decls: ModuleLevelDecl[],
     segmentUsage: Map<string, Set<string>>,  // segmentSymbolName -> set of variable names used
     rootUsage: Set<string>,                   // variable names used in root (non-segment) code
   ): MigrationDecision[]
   ```
   Decision tree (from RESEARCH.md Pattern 6):
   a. If `decl.isExported` AND used by any segment → `reexport` (MIG-03)
   b. If `decl.isExported` AND NOT used by any segment → `keep`
   c. If used by root code → `reexport`
   d. If used by more than one segment → `reexport` (MIG-02)
   e. If `decl.hasSideEffects` → `reexport` (MIG-04)
   f. If `decl.isPartOfSharedDestructuring` → `reexport` (MIG-05)
   g. If used by exactly one segment → `move` with `targetSegment` set (MIG-01)
   h. If not used by any segment → `keep`

5. Export `computeSegmentUsage()` helper:
   ```typescript
   export function computeSegmentUsage(
     program: any,
     extractions: Array<{ symbolName: string; argStart: number; argEnd: number }>,
   ): { segmentUsage: Map<string, Set<string>>; rootUsage: Set<string> }
   ```
   Walk the AST collecting all Identifier references. For each identifier, determine if it falls within any extraction's arg range (argStart..argEnd) — if so, attribute it to that segment. If it falls outside ALL extraction ranges, attribute it to root. Return both maps.
