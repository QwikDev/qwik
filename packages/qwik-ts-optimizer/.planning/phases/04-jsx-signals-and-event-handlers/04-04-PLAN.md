<objective>
Build the loop hoisting module that detects event handlers inside loops, hoists .w([captures]) above the loop, injects q:p/q:ps props for iteration variable access, and generates positional parameter padding.

Purpose: When event handlers are inside loops, Qwik's runtime needs the capture wrapping hoisted outside the loop so each iteration shares the same QRL reference. Loop variables are passed through q:p/q:ps positional parameters instead of captures.
Output: src/optimizer/loop-hoisting.ts module + comprehensive unit tests
</objective>

  <action>
    Create `src/optimizer/loop-hoisting.ts` with:

    1. `LoopContext` type:
       ```typescript
       interface LoopContext {
         type: 'map' | 'for-i' | 'for-of' | 'for-in' | 'while' | 'do-while';
         iterVars: string[];      // Iteration variable names (e.g., ['item'] for .map, ['i'] for for-i)
         loopNode: any;           // AST node of the loop
         loopBodyStart: number;   // Start offset of the loop body
         loopBodyEnd: number;     // End offset of the loop body
       }
       ```

    2. `detectLoopContext(node: any, source: string): LoopContext | null`
       Detection rules per RESEARCH.md (LOOP-05):
       - `.map(callback)` — CallExpression where callee is MemberExpression with property `map`, iterVar from callback params
       - `for (let i = 0; ...)` — ForStatement, iterVars from init declarators
       - `for (const item of ...)` — ForOfStatement, iterVars from left declarators
       - `for (const key in ...)` — ForInStatement, iterVars from left declarators
       - `while (...)` — WhileStatement (no iterVars)
       - `do { ... } while (...)` — DoWhileStatement (no iterVars)

    3. `HoistingPlan` type:
       ```typescript
       interface HoistingPlan {
         hoistedDecl: string;     // e.g., "const handler = q_handler_qrl.w([cart])"
         hoistInsertOffset: number; // Where to insert (before the loop)
         qrlRefName: string;       // e.g., "handler" (the name used inside the loop)
         originalQrlRef: string;   // e.g., "q_handler_qrl" (the original QRL reference)
       }
       ```

    4. `hoistEventCaptures(qrlRefName: string, captureNames: string[], loopCtx: LoopContext): HoistingPlan | null`
       - If captureNames is empty, return null (no hoisting needed)
       - Build hoisted declaration: `const {qrlRefName} = {originalQrlRef}.w([{captureNames.join(', ')}])`
       - The hoisted declaration goes BEFORE the loop (at `loopCtx.loopBodyStart` or before the entire loop statement)
       - Return the plan (actual insertion done by the pipeline in Plan 05)

    5. `findEnclosingLoop(node: any, ancestors: any[]): LoopContext | null`
       - Walk up the ancestor chain to find if this node is inside a loop
       - Return the nearest enclosing loop context, or null if not in a loop

    NOTE on nested loops (Pitfall 5 from RESEARCH):
    - Inner loop captures from outer loop are passed through `.w()` at the inner loop level
    - `.w()` must be placed where all captured variables are in scope
    - For nested loops, this means the hoisted decl goes inside the outer loop body but before the inner loop

    Extend `src/optimizer/loop-hoisting.ts`:

    1. `generateParamPadding(loopVarCount: number): string[]` — generate positional parameter names:
       - First 2 positions are always padding: `["_", "_1"]` (event param and context param)
       - Remaining positions are for loop variables
       - Return array like `["_", "_1"]` or `["_", "_1", "item"]`

    2. `buildQpProp(loopVars: string[]): { propName: string; propValue: string }` — generate the q:p or q:ps prop:
       - Single var: `{ propName: "q:p", propValue: varName }`
       - Multiple vars: `{ propName: "q:ps", propValue: "[" + sorted(vars).join(", ") + "]" }`
       - Sort alphabetically for q:ps (per RESEARCH.md Pattern 7)

    3. `LoopHoistResult` type:
       ```typescript
       interface LoopHoistResult {
         hoistedDecl: string | null;    // .w() declaration to hoist
         hoistOffset: number;            // Where to insert hoisted decl
         qpProp: { propName: string; propValue: string } | null;  // q:p or q:ps to add
         paramNames: string[];           // Updated paramNames for the segment
         flags: number;                  // Flag value (includes bit 2 for loop context)
       }
       ```

    4. `analyzeLoopHandler(extraction: any, loopCtx: LoopContext): LoopHoistResult`
       - Determine which loop variables the handler references
       - Build q:p/q:ps prop for referenced loop vars
       - Generate param padding for segment paramNames
       - Compute flags with loop bit (4) set
       - Plan .w() hoisting for non-loop captures

    Per RESEARCH.md Pattern 7 and Pitfall 4:
    - The handler segment's signature becomes `(_, _1, loopVar)` where `_` and `_1` are padding for the event and context parameters that the runtime provides
    - `q:p` carries a single loop variable value at runtime
    - `q:ps` carries multiple loop variables as a sorted array
    - The segment's `paramNames` metadata includes the padding + loop var names
    - The flags for the JSX element include bit 2 (4) when q:p/q:ps is present
