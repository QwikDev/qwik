    **3. Wire JSX pipeline into transform.ts:**

    After step 2a (capture analysis) and before step 3 (rewrite parent), add a JSX transformation phase:

    ```
    // Phase 4: JSX transformation
    // Only for JSX files (.tsx, .jsx)
    if (ext === '.tsx' || ext === '.jsx') {
      // The JSX transform operates on the parent module source text via magic-string.
      // It runs AFTER extraction (so $() closures are already identified as segments)
      // but BEFORE parent rewriting (so JSX nodes still exist in the source).
      //
      // Pipeline order within JSX transform:
      // a. Detect loop contexts for event handler hoisting
      // b. For each JSX element (bottom-up):
      //    i.   Process bind: props -> expand to value + handler
      //    ii.  Process event props -> rename to q-e:/q-d:/q-w:
      //    iii. Analyze signal expressions in prop values -> _wrapProp/_fnSignal
      //    iv.  Classify remaining props -> varProps/constProps
      //    v.   Process children
      //    vi.  Compute flags + key
      //    vii. Generate _jsxSorted/_jsxSplit call
      // c. Hoist _hf functions to module scope
      // d. Hoist .w() declarations above loops
      // e. Track needed imports (_jsxSorted, _Fragment, _wrapProp, _fnSignal, etc.)
    }
    ```

    The JSX transform result should provide:
    - Modified source text (via magic-string mutations)
    - List of imports needed (to be added by rewrite-parent)
    - Signal hoister declarations (to be prepended)
    - Loop hoisting declarations (to be inserted)

    **4. Update rewrite-parent.ts:**
    - Accept JSX transform imports (new symbols like `_jsxSorted`, `_Fragment`, `_wrapProp`, `_fnSignal`, `inlinedQrl`, `_val`, `_chk`, `_jsxSplit`, `_getVarProps`, `_getConstProps`) and add them to the import section
    - Handle `_Fragment` import from `@qwik.dev/core/jsx-runtime` (different source than other imports)
    - Insert signal hoister `_hf` declarations at module scope (before the first non-import statement)

    **5. Update segment-codegen.ts:**
    - Handle segments that have `paramNames` from loop hoisting (these were set by `analyzeLoopHandler`)
    - When a segment is an event handler in a loop, its body needs to accept `(_, _1, loopVar)` parameters — this may require modifying the segment body function signature

    **Key integration points:**
    - JSX transform must run on the ORIGINAL source text (before parent rewriting removes $() calls). However, the event handler closures are already extracted as segments. The JSX transform converts the JSX syntax while the extracted segment references (like `q_handler_qrl`) are injected by parent rewriting.
    - The ordering is: extract -> capture analysis -> JSX transform (on source) -> parent rewrite (uses JSX-transformed source + extraction info) -> segment codegen
    - Actually, reconsider: the parent rewriting in rewrite-parent.ts uses magic-string on the original source. The JSX transform also uses magic-string. They should share the SAME magic-string instance, or the JSX transform should run first and pass the modified source to parent rewriting.
    - RECOMMENDED APPROACH: Run JSX transform on the magic-string instance INSIDE rewrite-parent.ts. This way both JSX rewriting and extraction rewriting operate on the same magic-string, avoiding conflicts. Pass the JSX modules into rewriteParentModule as optional deps.
