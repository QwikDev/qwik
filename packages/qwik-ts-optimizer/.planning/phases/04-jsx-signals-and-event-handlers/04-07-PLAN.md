    This task wires loop detection and hoisting into the JSX pipeline. Loop hoisting is a structural transform that operates at a higher level than per-prop processing.

    **1. Add loop-hoisting imports to jsx-transform.ts:**
    ```typescript
    import { detectLoopContext, findEnclosingLoop, buildQpProp, type LoopContext } from './loop-hoisting.js';
    ```

    **2. Track loop context during AST walk in transformAllJsx():**

    The `walk()` call in transformAllJsx currently only processes JSXElement/JSXFragment in the `leave` callback. Add an `enter` callback to detect loop contexts:

    ```typescript
    const loopStack: LoopContext[] = [];

    walk(program, {
      enter(node: any) {
        const loopCtx = detectLoopContext(node, source);
        if (loopCtx) {
          loopStack.push(loopCtx);
        }
      },
      leave(node: any) {
        // Check if leaving a loop node
        if (loopStack.length > 0 && loopStack[loopStack.length - 1].loopNode === node) {
          loopStack.pop();
        }

        // Existing JSXElement/JSXFragment handling...
        // BUT now pass the current loop context (loopStack at top) to transformJsxElement
      }
    });
    ```

    **3. Pass loop context to transformJsxElement():**

    Add `loopCtx: LoopContext | null` parameter to transformJsxElement(). When loopCtx is not null:

    a. Set `inLoop = true` for flags computation (bit 2, value 4)
    b. For each event handler prop on an HTML element, check if it has captures that reference loop iteration variables
    c. If so, inject `q:p` or `q:ps` prop into constEntries:
       - Call `buildQpProp(loopCtx.iterVars)` to get the prop entry
       - Single var -> `"q:p": item`
       - Multiple vars -> `"q:ps": [var1, var2]` (sorted)

    **4. Track .w() hoisting needs in JsxTransformOutput:**

    Update `JsxTransformOutput` to include:
    ```typescript
    hoistingInsertions: Array<{
      insertBeforePos: number;  // Position before the loop node
      declaration: string;      // e.g., "const handler_qrl = q_handler.w([cart]);\n"
    }>;
    ```

    During the walk, when processing an event handler inside a loop:
    - Identify the QRL reference name for the handler (e.g., `q_Component_onClick_abc123`)
    - Identify which captures are loop variables vs outer-scope captures
    - The `.w([outerCaptures])` goes BEFORE the loop, the loop variable is passed via `q:p`/`q:ps`
    - Record the insertion point (loop node start position) and declaration text

    **5. Apply hoisting insertions in rewrite-parent.ts:**

    After the `transformAllJsx()` call (around line 307), if `jsxResult.hoistingInsertions` has entries:
    ```typescript
    for (const insertion of jsxResult.hoistingInsertions) {
      s.appendLeft(insertion.insertBeforePos, insertion.declaration);
    }
    ```

    **6. Handle the flags correctly:**

    When `loopCtx` is present, `computeFlags()` already supports `inLoop` parameter (bit 2, value 4). Pass `inLoop: loopStack.length > 0` to ensure flags include the loop bit.

    **IMPORTANT NOTES:**
    - Loop hoisting is complex. The .w() hoisting only applies when the event handler has captures from OUTSIDE the loop that need to be bound once (not per-iteration). Loop iteration variables use q:p/q:ps instead.
    - The actual segment body modification (adding _, _1 parameter padding) happens at the extraction/segment-codegen level, not in the JSX transform. The JSX transform's job is: (a) detect loop context, (b) inject q:p/q:ps props, (c) set the loop flag bit, (d) record .w() hoisting insertions.
    - If the integration proves too complex to get right in one pass (nested loops, cross-scope captures), focus on the simpler case first: single .map() loop with one event handler. The unit tests for loop-hoisting.ts already cover edge cases; the integration just needs to invoke the module correctly.
