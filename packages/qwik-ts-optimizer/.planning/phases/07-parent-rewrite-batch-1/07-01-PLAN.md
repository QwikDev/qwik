This causes AST comparison failures on all 24 Phase 7 snapshots and specifically is the ONLY issue for the 3 segment-strategy snapshots.

Output: Modified `rewrite-parent.ts` with unified import block assembly, and adjusted `removeUnusedImports` in `transform.ts` that preserves non-marker Qwik core specifiers.

**Current behavior (broken):** Step 5 builds `importStatements` (optimizer-added imports) and `s.prepend()` puts them before the body. But original user imports remain in their original positions in the body text (edited but not moved).

**Required behavior (from snapshot analysis):**

```
//
```

**Import ordering rule (derived from snapshot analysis):**
1. Optimizer-added imports -- each symbol on its own `import { symbol } from "source";` line. Order: the Qrl-suffixed callee imports in extraction order, then _noopQrl/_noopQrlDEV, then other optimizer symbols (_captures, _qrlSync, etc.), then JSX helpers (_jsxSorted, etc.), then Fragment.
2. User imports -- original import declarations with marker specifiers removed, in their original source order. Preserve original quote style. Non-marker specifiers from Qwik core imports are kept (e.g., `useStore`).

**Implementation approach:**

1. In Step 2 (marker removal), instead of using `s.overwrite()` on individual import declarations, REMOVE all import declarations from the magic-string body entirely (use `s.remove(node.start, end)` for each import declaration). Track the surviving user imports (those with at least one non-marker specifier remaining) in a `survivingUserImports: string[]` array. When rebuilding a user import with some markers removed, reconstruct using the ORIGINAL source's quote style (detect single vs double quotes from `node.source.raw`).

2. In Step 6 (assembly), build the preamble as:
   ```
   [...optimizerImportStatements, ...survivingUserImports, '//']
   ```
   Then QRL declarations, then `//`, then .s() calls (if inline), then `//`.

**Critical details:**
- When removing ALL specifiers from a user import (all are markers), the import is fully removed (no surviving entry).
- When removing SOME specifiers, reconstruct: `import { remaining, specifiers } from <original_quote_style>source<original_quote_style>;`
- Side-effect imports (`import 'module'`) are preserved in their original position, NOT moved to the preamble.
- Default imports and namespace imports should be preserved in the surviving user import line.
- The `//` separator between imports and QRL decls must appear exactly once, after all imports.
- Use `node.source.raw` to detect single vs double quotes (if raw starts with `'`, use single quotes; otherwise double).

Modify `removeUnusedImports()` in `transform.ts` to NOT remove import specifiers that were originally in the user's source code and are from Qwik core packages -- even if they appear unused after extraction.

**Why:** The Rust optimizer keeps non-marker specifiers from Qwik core imports even when the identifier is no longer referenced in the parent body. For example, `import { useStore } from '@qwik.dev/core'` stays in `example_1` parent output even though `useStore` only appears inside the extracted segment body.

**Current behavior:** `removeUnusedImports()` removes ANY specifier whose local name has zero references in the remaining body, including Qwik core specifiers like `useStore`.

**Required change:** After Task 1 reassembles imports in the preamble, the user imports are already correctly positioned. `removeUnusedImports()` should skip removal of specifiers from Qwik packages (those whose source starts with `@qwik.dev/` or `@builder.io/qwik`).

**Implementation:**

In `removeUnusedImports()`, around line 230 where `unreferencedSpecs` is computed, add a filter:

```typescript
const unreferencedSpecs = importSpecs.filter(
  (spec) => {
    if (referencedNames.has(spec.localName)) return false; // referenced, keep
    // Check if this import is from a Qwik package -- if so, keep it
    const source = spec.node.source?.value ?? '';
    if (QWIK_IMPORT_PREFIXES.some(prefix => source.startsWith(prefix))) {
      return false; // Qwik package import, preserve even if unused
    }
    return true; // non-Qwik, unreferenced -> remove
  }
);
```

The `QWIK_IMPORT_PREFIXES` constant is already defined at line 125-128:
```typescript
const QWIK_IMPORT_PREFIXES = ['@qwik.dev/', '@builder.io/qwik'];
```

**Important:** This means the existing comment at lines 176-178 about intentionally NOT skipping Qwik imports needs to be updated. The old logic was correct for removing marker specifiers (like `component$`), but now those markers are already removed by `rewriteParentModule()` Task 1. What remains in the preamble are the non-marker user specifiers, which should be preserved.

Actually, re-checking: after Task 1 rebuilds imports with markers removed, the surviving user imports in the preamble should only have non-marker specifiers. The `removeUnusedImports` re-parse would see these. We need to make sure those are not removed.

The safest approach: for Qwik package imports, only remove specifiers that end with `$` (marker functions) or match known optimizer-added symbols like `qrl`, `qrlDEV`, `_noopQrl`, `_noopQrlDEV`, `_qrlSync`, etc. Everything else from Qwik packages is a user import and should be preserved.

Simpler approach (recommended): Just skip ALL Qwik package imports from unused-import cleanup entirely. The rewriter already handles marker removal in Step 2. Anything left in a Qwik import after Step 2 is intentional.
