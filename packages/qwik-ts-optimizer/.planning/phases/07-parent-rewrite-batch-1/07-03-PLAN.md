Fix three segment-strategy parent output issues:

**A. Unused binding removal (example_1):**
In rewrite-parent.ts Step 4, when rewriting a named marker call site like `const renderHeader2 = component$(...)`, check if the variable (`renderHeader2`) is:
1. Not exported (`export const` absent)
2. Not referenced anywhere else in the module body (after extraction)

If both true, remove the entire `const varName = ` prefix from the variable declaration, leaving just the bare call expression `componentQrl(q_var)`. This matches the Rust optimizer which strips unused variable bindings.

Implementation approach: After Step 4 call site rewriting, scan the magic-string output for `VariableDeclaration` nodes where the init was a marker call. For each such declaration, if the variable is not exported and not used elsewhere in the parent body (check via string search for the var name as a word boundary), remove the `const/let/var varName = ` prefix. The variable declaration node positions are available from the original AST parse.

**B. User import preservation (example_functional_component):**
In transform.ts `removeUnusedImports()`, when processing Qwik core imports (source matches `@qwik.dev/core` or `@builder.io/qwik`), preserve the ORIGINAL import specifiers (excluding markers) even if their identifiers are no longer referenced in the parent body after extraction. The Rust optimizer behavior: it removes only the $-suffixed marker specifiers from Qwik core imports, but keeps all other specifiers from the same import statement.

Specifically: In `removeUnusedImports()`, when an import source is a Qwik core package, skip the "is this specifier referenced?" check. Only remove specifiers that are $-suffixed marker functions. Keep `useStore`, `component`, `$`, etc. if they were in the original import.

NOTE: This must be carefully tested against the 16 already-passing snapshots. The Phase 07-01 SUMMARY notes that blanket Qwik import preservation caused regressions. The key difference: preserve specifiers from the ORIGINAL user import statement that are NOT markers, but still clean up specifiers that are markers. The issue in Plan 01 was trying to preserve ALL Qwik specifiers -- we only need to preserve non-marker specifiers from user-written imports.

**C. Signal wrapping in $() callback body (example_missing_custom_inlined_functions):**
This snapshot's expected output has `_wrapProp(state, "count")` inside a `$()` callback return body. The actual output has bare `state.count`. This means the JSX transform within the `$()` callback (which is an extraction argument range) is NOT applying signal wrapping.

The issue: Step 4c JSX transform uses `skipRanges` to skip extraction argument ranges. But the JSX within those argument bodies SHOULD be transformed -- the skip ranges should only prevent overwriting the call site text, not the JSX inside.

Investigate the skipRanges logic: the intent is to skip regions already rewritten by Step 4, but the JSX inside extraction args needs to be processed. For segment strategy, the args are replaced with QRL variable names, so there's no JSX left. But for cases where the body text includes JSX that feeds into a different extraction's return value, the signal analysis needs to handle it.

Actually, re-read the expected output carefully: `return $(()=>/*#__PURE__*/ _jsxSorted("div", null, null, _wrapProp(state, "count"), 3, "u6_0"))`. The `$()` call is KEPT (not extracted to a QRL) because it's a custom inlined function's argument. So the JSX inside should be transformed by Step 4c. Check if the skipRanges are too aggressive -- they should only skip ranges for EXTRACTED calls, not for all `$()` calls.

Focus on fixing the variable migration first.

**C. Destructuring optimization (example_optimization_issue_3542):**
The expected output has `_rawProps` pattern: component functions receive `_rawProps` instead of destructured `{ctx, atom, ...}`, and inside the body, `ctx` becomes `_rawProps.ctx`, `atom` becomes `_rawProps.atom`.

This is the "props destructuring optimization" from the Rust optimizer. When a component's arrow function has destructured parameters, the Rust optimizer:
1. Replaces the destructured pattern with `_rawProps`
2. Rewrites all references to destructured names as `_rawProps.propName`

This is a significant feature. For this plan, implement a basic version in the `.s()` body transformation:
- In `transformSCallBody()`, when the extraction's parameter is a destructuring pattern, replace it with `_rawProps` and rewrite body references.
- The extraction result should already have the parameter pattern info.

Check what data is available on the extraction: `ext.paramNames` or similar. If the destructuring info isn't captured during extraction, this task may need to add it.
