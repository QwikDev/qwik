Implement the hoist-strategy output pattern. The Rust optimizer's hoist output differs from inline:

**Inline pattern (current, correct for inline strategy):**
```javascript
q_Component_symbolHash.s(()=>{
    // transformed body with JSX, signals, etc.
});
```

**Hoist pattern (needed for hoist strategy):**
```javascript
const Component_symbolHash = (params)=>{
    // transformed body with JSX, signals, etc.
};
q_Component_symbolHash.s(Component_symbolHash);
```

Key differences:
1. Body is extracted into a **named const** using the extraction's symbolName (not the QRL var name)
2. The `.s()` call takes **just the variable name** (not the arrow function)
3. The const declaration appears AFTER the QRL declarations block (`//` separator) and BEFORE the `.s()` call
4. The body in the const function has JSX transpiled (reuse Plan 04's JSX body transformation)

**Implementation in rewrite-parent.ts Step 5c:**

When building .s() calls, detect if the entry strategy is `hoist` (not `inline`). The `inlineOptions` parameter tells us `inline: true` for both inline and hoist strategies currently. Need to differentiate:
- Check the entryStrategy type in transform.ts and pass it through to rewriteParentModule.
- Add an `entryType` field to InlineStrategyOptions: `'inline' | 'hoist'`.

For hoist strategy:
1. After transformSCallBody (which applies JSX transpilation from Plan 04), instead of building `qrlVar.s(transformedBody)`:
   a. Create a const declaration: `const {symbolName} = {transformedBody};`
   b. Create a simple .s() call: `qrlVar.s({symbolName});`
2. Push the const declaration to a separate array (hoistConstDecls)
3. Push the .s() call to sCalls
4. In Step 6 preamble assembly, place hoistConstDecls AFTER the `//` + QRL decls block and BEFORE the `//` + .s() calls block

**In inline-strategy.ts:**
Add `buildHoistConstDecl(symbolName: string, body: string): string` that returns `const ${symbolName} = ${body};`
Add `buildHoistSCall(qrlVarName: string, symbolName: string): string` that returns `${qrlVarName}.s(${symbolName});`

**Preamble structure for hoist:**
```
[optimizer imports]
[user imports]
//
[hoisted signal declarations - _hf0, _hf0_str, etc.]
//
[QRL declarations - const q_xxx = _noopQrl("xxx")]
//
[const function declarations - const Component_hash = (params)=>{...}]
[.s() calls - q_xxx.s(Component_hash)]
[additional .s() calls]
//
[body - exports, etc.]
```

Study the expected snapshot output to determine the exact ordering of these sections.

**Testing approach:**
Start with `example_mutable_children` -- it's the simplest hoist snapshot (needs _wrapProp but not full JSX transpilation). Then verify `example_immutable_function_components` and `example_derived_signals_div`.
