**Fix 1: issue_476 -- No-extraction passthrough**

When `extractSegments()` returns zero extractions and the entry strategy is `segment`, the source should be passed through with only the `//` separator added. Currently, import stripping logic may remove non-marker imports even when there are no markers.

In transform.ts, after extraction, if `extractions.length === 0`:
- Return the source as the parent module (with `//` separator if needed)
- Do NOT run import stripping or unused import removal
- Preserve all original imports unchanged

The expected output for issue_476 is essentially the input with a `//` separator between imports and body.

**Fix 2: should_ignore_null_inlined_qrl -- Pre-existing inlinedQrl calls**

When input contains `inlinedQrl(null, 'some_hash')`, these are already-optimized QRL calls that should NOT be treated as marker calls for extraction.

Check how marker detection works -- `inlinedQrl` should not be in the marker detection map. If it IS being detected as a marker, exclude it. The key is that `inlinedQrl` is not a `$()` marker -- it's an already-processed output.

If the issue is that the unused binding (`const foo = inlinedQrl(null, ...)`) is being stripped, ensure the unused binding removal logic in Step 4a of rewrite-parent.ts skips bindings whose init is an `inlinedQrl()` call. The expected output converts `const foo = inlinedQrl(null, ...)` to just `inlinedQrl(null, ...);` (drops the unused `const foo =`).

Study the snapshot carefully to understand what transformation is expected.

The hoist-to-const infrastructure (buildHoistConstDecl, buildHoistSCall) exists in inline-strategy.ts and is used in rewrite-parent.ts for `entryType: 'hoist'`. For `entryType: 'inline'`, the code currently always uses `buildSCall` which inlines the body directly in `.s(() => { ... })`.

Study the two target snapshots to determine when inline strategy should use hoist-to-const instead:

1. `inlined_qrl_uses_identifier_reference_when_hoisted_snapshot`: The expected output has `const App_component_ckEPmXZlub0 = () => { ... }; q_App_component.s(App_component_ckEPmXZlub0);` -- the body is hoisted to a const and referenced by name.

2. `should_not_generate_conflicting_props_identifiers`: Similar pattern with `_captures` usage.

In the Step 5c dispatch in rewrite-parent.ts, add a condition: when using inline strategy AND the extraction qualifies for hoist-to-const (determine from snapshot analysis -- likely when body is a component/complex or uses `_captures`), use `buildHoistConstDecl` + `buildHoistSCall` instead of `buildSCall`.

The exact qualification rule needs to be derived from snapshot comparison. Start with: always hoist-to-const for inline strategy (same as hoist strategy behavior). If that causes regressions, narrow the rule.
