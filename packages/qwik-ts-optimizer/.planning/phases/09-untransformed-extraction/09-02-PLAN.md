1. Create src/optimizer/input-repair.ts with a `repairInput(source: string): string` function:

   The function attempts to parse the input with oxc-parser. If the parse succeeds (program.body.length > 0), return the input unchanged. If program.body is empty AND there are errors, apply targeted repairs:

   **Repair pattern A (example_3 -- stray closing paren):**
   The input has `export const App = () => { ... });` -- the `)` before `};` is unmatched. The pattern is: an arrow function body block `=> { ... }` that has a stray `)` before the closing `}` or `};`.
   
   Strategy: When parse fails with empty body, try removing unmatched `)` characters. For each `)` in the source, check if removing it makes the parse succeed. This is a brute-force approach but safe since we only apply it when the original parse already fails.
   
   Simpler approach: Count parens. If there are more `)` than `(`, identify the extra `)` candidates (ones not preceded by a matching `(`) and try removing them one at a time.

   **Repair pattern B (example_immutable_analysis -- JSX text containing arrow syntax):**
   The input has `[].map(() => (` appearing as raw text in JSX children (not wrapped in `{}`). This is invalid JSX that SWC somehow handles.
   
   Strategy: When parse fails, look for `>` followed by text that contains `=>` or other JS operators before the next `<`. These are likely meant to be JSX expression containers. Wrap them in `{` and `}`.
   
   Actually, check the EXACT input for example_immutable_analysis more carefully first. Read the full input to understand the exact parse failure. The issue may be simpler than described.

   **Important:** repairInput should be conservative. Only apply repairs when parseSync returns empty body with errors. If the first repair makes parse succeed, stop. If no repair works, return original source unchanged.

2. In extract.ts, call repairInput at the top of extractSegments:
   ```typescript
   import { repairInput } from './input-repair.js';
   // At start of extractSegments:
   const repairedSource = repairInput(source);
   const { program } = parseSync(relPath, repairedSource);
   ```
   
   IMPORTANT: When repairInput modifies the source, the character positions in the AST will be relative to the REPAIRED source, not the original. Since magic-string operations in rewrite-parent.ts operate on the original source, we need to track position offsets. For simplicity, if the repair only REMOVES characters, the position mapping is straightforward -- but this adds complexity.
   
   Alternative approach: Instead of modifying source in extract.ts, apply the repair in transformModule BEFORE calling extractSegments, and use the repaired source throughout the entire pipeline (extraction, capture analysis, parent rewriting). This is cleaner because all position references are consistent.

3. In transform.ts, apply repairInput before extractSegments call:
   ```typescript
   import { repairInput } from './input-repair.js';
   // In transformModule, after computing relPath:
   const repairedSource = repairInput(source);
   // Use repairedSource instead of source for extractSegments and magic-string
   ```
   
   This way the MagicString operates on the repaired source, and all positions from the AST are consistent with what magic-string sees.
