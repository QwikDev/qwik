When captures will be delivered as function parameters (paramNames includes the captured variables alongside `_` placeholders), the captures flag must be false. This matches Rust's `captures: !h.data.scoped_idents.is_empty()` where scoped_idents is empty when captures become params.

The key insight from snapshot analysis: when `ext.paramNames` contains captured variable names (not just `_` placeholders), those captures are delivered via parameter injection, not via the `_captures` mechanism. In that case, `captures` should be `false`.

**Detection heuristic:** After the migration filtering at line 871-872, check if `ext.captureNames` has entries AND `ext.paramNames` already includes those capture names. If all remaining captureNames are present in paramNames, set `captures = false` because they're delivered as params, not via _captures.

However, looking at the snapshots more carefully:
- lib_mode_fn_signal: paramNames `["_", "_1", "count"]`, captureNames likely `["count"]` -> captures should be false
- should_extract_single_qrl_2: paramNames `["_", "_1", "results"]`, captureNames likely `["results"]` -> captures should be false

The simpler approach: When `ext.paramNames.length > 0` and all entries in `ext.captureNames` appear in `ext.paramNames`, set `ext.captures = false` because the captures are delivered via parameter injection.

Add after line 872:
```typescript
// Reconcile captures with paramNames: when captures are injected as
// function parameters (Rust's _auto_ pattern), the captures flag is false
// because scoped_idents is empty -- the captured variables become formal params.
if (ext.captures && ext.paramNames.length > 0) {
  const paramSet = new Set(ext.paramNames);
  const allCapturesInParams = ext.captureNames.every(name => paramSet.has(name));
  if (allCapturesInParams) {
    ext.captures = false;
  }
}
```

IMPORTANT: This must NOT affect cases where captures use the _captures/.w() mechanism (like some segments in example_qwik_router_client where captures is expected to be true). The heuristic `allCapturesInParams` correctly handles this -- if captureNames are NOT in paramNames, captures stays true.

NOTE: If paramNames is not yet populated at this point in the pipeline, you may need to investigate when paramNames gets set. Read the capture-analysis.ts output to understand the flow. The captures reconciliation should happen AFTER both captureNames and paramNames are finalized.
