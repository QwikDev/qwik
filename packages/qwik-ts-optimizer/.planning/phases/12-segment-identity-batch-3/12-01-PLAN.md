After capture analysis runs (around line 548 in transform.ts, after the `analyzeCaptures` call and initial reconciliation), add loop-aware capture classification:

1. For each extraction that is an eventHandler (ctxKind === 'eventHandler'), detect if it is inside a loop by re-parsing the original source AST and walking ancestors. The extraction's `argStart`/`argEnd` positions can be used: walk the AST to find the node at `argStart`, collect ancestors, and call `findEnclosingLoop`.

   Approach: After the existing `parseSync` of the input code (variable `program` at line 460), walk the AST once to build a map of extraction positions to their enclosing loop contexts. For each extraction, find the AST node whose range contains `[extraction.argStart, extraction.argEnd]` and walk its ancestors to find loops.

   Simpler approach: Use the already-parsed `program` AST. Walk it with `walk()` from oxc-walker, tracking a `loopStack` (push on enter when `detectLoopContext` returns non-null, pop on leave). When entering a node whose range matches an extraction's `[callStart, callEnd]`, record the current loopStack state for that extraction.

2. For extractions inside a loop:
   a. Collect ALL variables from enclosing loops' `iterVars` arrays. Also collect block-scoped declarations inside the loop body by scanning for VariableDeclarations in the loop body range.
   b. Partition `captureNames` into two groups:
      - **loopLocalVars**: captureNames that appear in any enclosing loop's iterVars OR are declared inside a loop body (block-scoped). These are delivered via q:p params.
      - **crossScopeCaptures**: captureNames NOT in any enclosing loop scope. These need _captures mechanism.
   c. Set `extraction.paramNames = generateParamPadding(loopLocalVars)` (from loop-hoisting.ts).
   d. Set `extraction.captureNames = crossScopeCaptures` (sorted).
   e. Set `extraction.captures = crossScopeCaptures.length > 0`.

3. For NESTED loops (extraction inside inner loop captures var from outer loop):
   - The extraction is inside the INNER loop
   - A captured variable declared in the OUTER loop's iterVars is a cross-scope capture (captures: true)
   - A captured variable declared in the INNER loop's iterVars is loop-local (delivered via q:p)
   - Walk ALL enclosing loops from innermost outward. Only vars from the IMMEDIATE enclosing loop (the one whose body directly contains the JSX element) are loop-local. Vars from outer loops are cross-scope.

   Determining the "immediate" loop: The extraction's parent segment contains the JSX element. The loop that directly wraps the JSX element call site is the immediate loop. Use the innermost loop in the loopStack.

4. Block-scoped variable detection: Variables like `const index = i + 1` inside `.map((val, i) => { const index = ... })` are loop-local. To detect these, for each enclosing loop, collect all VariableDeclaration binding names within `[loopBodyStart, loopBodyEnd]`. These count as loop-local vars for the immediate loop.

   Implementation: Walk the AST within the loop body range and collect all VariableDeclaration identifiers. Use `collectScopeIdentifiers` on the loop body node or manually walk declarations in the loop body range.

The logic:
1. Check if `extraction.paramNames.length > 0` AND the paramNames include padding (starts with `["_", "_1", ...]`).
2. If so, the bodyText currently has the original function signature (e.g., `() => console.log(index)` or `(event) => handleClick(event)`). Rewrite it to include the padding parameters.
3. Rewriting approach:
   a. Find the parameter list in bodyText. For arrow functions: text between `(` and `)` before `=>`. For function expressions: text between `(` and `)` after `function`.
   b. Replace the parameter list with the full `extraction.paramNames` joined by `, `.
   c. Example: bodyText `() => console.log(index)` with paramNames `["_", "_1", "index"]` becomes `(_, _1, index) => console.log(index)`.
   d. Example: bodyText `(event) => handleClick(event)` with paramNames `["_", "_1", "item"]` and the original param `event` is NOT in paramNames — this case means the original params are kept and loop vars are appended. Actually, looking at the snapshots, the original params are REPLACED entirely by paramNames. The Rust optimizer computes paramNames to include ALL needed params.

   Specifically for the Rust pattern:
   - Event handler with NO original params capturing loop var `index`: paramNames = `["_", "_1", "index"]`, bodyText: `() => console.log(index)` -> `(_, _1, index) => console.log(index)`
   - Event handler with original params: the original params should remain, and loop vars are appended. But looking at snapshots, the paramNames IS the full list. So just replace the params entirely.

4. Implementation:
   ```typescript
   // After body text processing, before export line
   if (extraction.paramNames.length >= 2 && 
       extraction.paramNames[0] === '_' && 
       extraction.paramNames[1] === '_1') {
     // This is a loop-context segment — rewrite function signature
     bodyText = rewriteFunctionSignature(bodyText, extraction.paramNames);
   }
   ```

5. `rewriteFunctionSignature(bodyText, paramNames)` function:
   - For arrow functions: find the opening `(` and closing `)` before `=>`, replace contents with `paramNames.join(', ')`.
   - For function expressions: find the opening `(` and closing `)` after `function`, replace contents.
   - Edge case: arrow with no parens (single param like `x => x`): wrap with parens and replace.
   - Use `findArrowIndex` (already exists in segment-codegen.ts) to locate the arrow, then scan backwards to find the param list.
