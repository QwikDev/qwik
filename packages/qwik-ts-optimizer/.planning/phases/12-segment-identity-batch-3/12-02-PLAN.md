**Part A: .w() hoisting in parent segment body (transformSCallBody)**

In `transformSCallBody()` (rewrite-parent.ts), when a child extraction has `captures: true` AND is inside a loop context (its paramNames starts with `["_", "_1", ...]`), the parent segment body needs a `.w([captureNames])` hoisting declaration inserted BEFORE the loop containing the child.

Currently (line ~479 and ~491), `.w()` is appended inline to the QRL reference. For loop cross-scope captures, the pattern is different — a separate `const` declaration:
```javascript
const childSymbol = q_childSymbol.w([outerVar]);
```
This declaration goes inside the parent body, positioned just before the inner loop where the child is used.

The existing `.w()` logic at line 479/491 already handles inline `.w()` for the default case. For loop-context children, instead of appending `.w()` to the QRL ref in the JSX, generate a standalone `const` declaration and use the unqualified name in the JSX.

Implementation approach:
1. In `transformSCallBody`, when processing children, check if the child has paramNames starting with `["_", "_1"]` AND `captures: true`.
2. If so, instead of appending `.w([...])` to the QRL reference inline, generate:
   ```
   const {childSymbolName} = q_{childSymbolName}.w([{captureNames.join(', ')}]);
   ```
3. Insert this declaration before the loop body in the transformed text.
4. In the JSX element, use the hoisted variable name (without q_ prefix) instead of the q_ QRL reference.

**Part B: q:p placement correction**

In `jsx-transform.ts` at line 819-828, q:p is currently always pushed to `constEntries`. The Rust optimizer places q:p in `varEntries` when the event handler for that element has `.w()` captures (cross-scope captures from an outer loop).

Change the q:p injection logic:
1. Check if any event handler prop on this element has captures (cross-scope, meaning the corresponding extraction has `captures: true`).
2. If YES: push q:p to `varEntries` instead of `constEntries`.
3. If NO: keep q:p in `constEntries` (current behavior).

The challenge: at JSX transform time, we don't have direct access to extraction metadata. The q:p placement decision may need to be determined by whether the element has a `.w()`-hoisted handler reference vs a direct q_ reference. Consider passing additional context about which QRL refs have captures, or making the decision based on available information in the transform context.

Simpler approach: Since q:p placement in varEntries only matters when the handler has cross-scope captures (.w() hoisting), and Plan 01 already classifies this, pass a flag or check the handler reference name. If the handler is a hoisted `const x = q_x.w([...])` variable (not a q_ prefixed ref), put q:p in varEntries.

**Part C: Component element event name preservation**

For `should_not_transform_events_on_non_elements`: when a component element (uppercase tag like `<CustomComponent>`) has an event handler like `onClick$`, the segment body should keep `onClick$={q_ref}` NOT transform it to `q-e:click`. This is because component elements are NOT HTML elements and don't use the q-e: prefix.

In `jsx-transform.ts`, the event prop transformation should check if the tag is a component (uppercase first char or contains a dot). If so, keep the original event name.

Check: This may already be partially handled (Phase 8 decision: "Component elements keep untransformed event names"). Verify the current behavior and fix if the segment body is not preserving component event names.

**Part A: Signal hoisted function deduplication**

In `SignalHoister` (signal-analysis.ts), when two signal expressions produce identical hoisted functions (same accessor pattern like `(p0) => p0.value.id`), they should share the same `_hf` declaration instead of generating separate `_hf0` and `_hf1`.

Implementation:
1. In the `SignalHoister` class, maintain a deduplication map: `Map<string, string>` mapping function body text to the existing `_hf` variable name.
2. Before creating a new `_hf` entry, check if an identical function body already exists in the map.
3. If it does, reuse the existing `_hf` variable name.
4. If not, create a new one and add to the map.

This affects the `should_transform_nested_loops` snapshot where `row.value.id` and `item.value.id` use the same accessor pattern `(p0) => p0.value.id`.
