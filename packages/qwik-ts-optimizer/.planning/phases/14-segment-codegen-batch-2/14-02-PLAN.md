  <action>
1. **TS enum value inlining in segments:** The approach depends on what oxc-transform can do with isolated segment body text. Two strategies:
   - **Strategy A (preferred):** Before extracting segment bodies, run oxc-transform on the FULL parent source with TS enum transpilation enabled. This replaces `enum Thing { A, B }` with IIFE patterns and inlines `Thing.A` -> `0` throughout the file. Then extraction proceeds on the already-transpiled source.
   - **Strategy B (fallback):** Build an enum-value map from parent AST (`enum Thing { A=0, B=1 }` -> `{Thing: {A: '0', B: '1'}}`), then do string replacement in segment bodies: `Thing.A` -> `0`.
   
   Try Strategy A first in `transform.ts`: if `transpileTs` is true, run `oxcTransformSync` with full TypeScript transpilation (not just type removal) on the source BEFORE the extraction pipeline. This should handle enums in both parent and segment output.

2. **Parent `let` vs `var` for enum declarations:** The research notes that oxc-transform produces `let` for transpiled enums while the Rust optimizer produces `var`. After oxc-transform runs on the parent, post-process to replace `let Thing = function(Thing)` with `var Thing = function(Thing)` for enum IIFE patterns. A targeted regex replacement should work: `/\blet\s+(\w+)\s*=\s*function\s*\(\1\)/g` -> `var $1 = function($1)`.

3. **example_ts_enums_no_transpile:** When `transpileTs` is false, enum declarations should remain as-is but the segment should still import the enum type. Verify this case doesn't break.

4. Strip `@qwik-disable-next-line` comments from segment bodies. Add a simple regex removal step in `generateSegmentCode` body processing: `bodyText = bodyText.replace(/\/\*\s*@qwik-disable-next-line\s+\w+\s*\*\/\s*\n?/g, '')`.
