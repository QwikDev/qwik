**Fix 1: Unconditional segment JSX (transform.ts ~line 1499)**
The current gate `(shouldTranspileJsx && (ext.extension === '.tsx' || ext.extension === '.jsx' || isJsx))` prevents segment JSX transpilation when `transpileJsx: false`. Remove the `shouldTranspileJsx` condition for segments only. Segments are standalone modules and MUST always have JSX transpiled regardless of the option. Change to:

**Fix 2: Pre-scan spread for bind gate (jsx-transform.ts processProps)**
Currently at line 646, `isBindProp(propName) && !hasSpread` uses `hasSpread` which only becomes true AFTER a spread is encountered in the loop. If a bind prop appears BEFORE a spread in the attributes, `hasSpread` is false and bind gets incorrectly desugared.

Add a pre-scan before the main attribute loop in processProps (after the early return at line 579):
```typescript
// Pre-scan for spreads so bind gate works regardless of attribute order
const hasSpreadAttr = attributes.some(a => a.type === 'JSXSpreadAttribute');
```

Then change line 646 from `!hasSpread` to `!hasSpreadAttr`:
```typescript
if (isBindProp(propName) && !hasSpreadAttr) {
```

When `hasSpreadAttr` is true and a bind prop is encountered, keep it as-is in varEntries:
```typescript
} else if (isBindProp(propName) && hasSpreadAttr) {
  // In spread/split mode, bind props pass through un-desugared into varEntries
  varEntries.push(`"${propName}": ${valueText}`);
  continue;
}
```
Note: the prop name must be quoted (e.g., `"bind:value"`) matching Rust AST key style.

**Fix _auto_ re-export suppression:**

The issue: variables like `export const useSecretAction = globalActionQrl(...)` or `export const PANELS = [...]` are already directly exported, but the migration logic still emits `export { useSecretAction as _auto_useSecretAction }` re-exports.

In variable-migration.ts, find where the `reexport` decision is made. Currently at ~line 344: `if (decl.isExported && usedByAnySegment)` the decision is `reexport`. But when a variable is already exported with `export const/let/var`, segments can import it directly by name -- no _auto_ prefix needed.

The fix depends on how rewrite-parent.ts generates the re-exports. Read the re-export generation code in rewrite-parent.ts and trace the flow:

1. In variable-migration.ts: When `decl.isExported` is true and the variable is used by a segment, the decision should still be `reexport` BUT with a flag indicating "already exported, no _auto_ needed" (or a separate decision like `keep` that still makes it available to segments).

2. In rewrite-parent.ts: Where `_auto_` re-export lines are generated, check if the variable is already exported. If so, skip the `_auto_` re-export line entirely -- the variable is already importable from the parent module.

The cleanest approach: in the migration decision, when `isExported` is true AND the variable is used by segments, set `action: 'keep'` (or a new action like `'already-exported'`) instead of `'reexport'`. Then in rewrite-parent.ts, skip `_auto_` generation for these. In segment-codegen.ts, the import should use the original name directly (e.g., `import { PANELS } from "./test"` not `import { _auto_PANELS as PANELS } from "./test"`).

Verify that `should_work`, `should_wrap_store_expression`, and `ternary_prop` no longer have spurious _auto_ re-exports.
