JsxKeyCounter needs to accept a key prefix in its constructor, computed from the srcDir-relative file path. Look at how the optimizer's `qwikHash` function works (in the siphash module) and how the Rust optimizer derives the key prefix. The key prefix is likely the first 2 characters of a base36-encoded hash of the relative file path.

Study the test infrastructure to find how `support_windows_paths` passes its file path. Then:
1. Add a `prefix` parameter to JsxKeyCounter constructor
2. Compute the prefix from the relative file path using the same hash as Rust (look at `new_jsx_dev_key` in Qwik Rust source, or reverse-engineer from "u6" = hash("test.tsx") and "KD" = hash("components/apps/apps.tsx"))
3. Thread the prefix through from transform.ts where the JsxKeyCounter is created
4. Verify: `support_windows_paths` now produces "KD_0" keys

**Fix 2: For-of loop variable capture promotion (capture-analysis.ts or loop-hoisting.ts)**

The `should_transform_handler_in_for_of_loop` snapshot expects `(_, _1, val) => console.log(val)` where `val` is the for-of iterator variable promoted to paramNames position 2.

Fix: Ensure for-of iterator variables are treated the same as for-loop variables for capture promotion to paramNames.

In variable-migration.ts, when building the migration decision for `a`, check if its declaration is a destructuring of an imported value.
