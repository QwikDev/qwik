  <action>
Fix `computeSegmentUsage` in variable-migration.ts to match SWC's behavioral rules. Two changes needed:

**Change 1 -- Filter locally-declared identifiers from segment usage:**
Add a helper function `collectLocalDeclarations(program, start, end)` that walks all nodes within a given range and collects identifiers that are declaration sites:
- `VariableDeclarator.id` -- collect all binding names from the pattern
- `FunctionDeclaration.id` -- the function name
- `CatchClause.param` -- the catch parameter
- `ArrowFunctionExpression.params` / `FunctionExpression.params` -- function parameters

In `computeSegmentUsage`, for each extraction, compute the local declarations within `argStart..argEnd`. Then when attributing an identifier to a segment, skip it if it is in that extraction's local declarations set. Use the existing `collectBindingNames` helper (already exported in the file) for pattern decomposition.

**Change 2 -- Filter declaration-site identifiers from rootUsage:**
When an identifier is outside all extraction ranges (root scope), check if it is the binding name of a declaration statement (i.e., the identifier appears as `VariableDeclarator.id`, `FunctionDeclaration.id`, or `ClassDeclaration.id` at the top level). Do NOT add these to rootUsage. SWC's `build_main_module_usage_set` explicitly skips `Stmt::Decl` items.

Implementation approach: Before the main walk, pre-compute two things:
1. A `Map<string, Set<string>>` of extraction-local declarations (key = symbolName)
2. A `Set<number>` of declaration-site identifier positions at the root level (walk program.body top-level statements, collect the `.start` positions of all binding name identifiers in declarations)

Then in the walk callback:
- For segment-attributed identifiers: skip if name is in that extraction's local declarations set
- For root-attributed identifiers: skip if position is in the declaration-site positions set

This fixes MIGR-01 (move vs reexport) and MIGR-02 (spurious _auto_ exports from shadowed vars).

<task type="auto">
  <name>Task 2: Ensure moved declarations carry import dependencies into segments</name>
  <files>src/optimizer/transform.ts, src/optimizer/segment-codegen.ts</files>
  <read_first>
    - src/optimizer/transform.ts (lines 1940-2000, the captureInfo wiring section)
    - src/optimizer/segment-codegen.ts (the movedDeclarations handling)
    - swc-reference-only/code_move.rs (lines 100-220 for segment import generation)
  </read_first>
  <action>
When a declaration is moved into a segment, the segment needs to import the identifiers that the moved declaration depends on. Currently `captureInfo.movedDeclarations` is just a `string[]` of declaration text. The segment codegen inserts these text strings but does not add the corresponding imports.

**Change 1 -- Expand movedDeclarations to carry dependency info:**
In `SegmentCaptureInfo` (segment-codegen.ts), change `movedDeclarations` from `string[]` to `Array<{ text: string; importDeps: Array<{ localName: string; importedName: string; source: string }> }>`. This is a breaking interface change so update all consumers.

**Change 2 -- Compute import dependencies for moved declarations:**
In transform.ts, in the section that populates `captureInfo.movedDeclarations` (around line 1978-1986), for each moved declaration:
1. Collect all identifiers referenced in the declaration text (walk the declaration's AST node range)
2. For each referenced identifier, check if it matches an import from `originalImports`
3. If yes, add that import to the moved declaration's `importDeps`

Use the existing `originalImports` array (already available in scope) which has `{ localName, importedName, source }` entries.

**Change 3 -- Emit import deps in segment codegen:**
In segment-codegen.ts, where movedDeclarations are inserted into the segment output, also add the import dependencies. Before the existing import dedup logic, merge the moved-declaration import deps into the segment's import set. Use the same dedup approach already used for other imports (check if the import line already exists in `parts`).

This fixes MIGR-03 (destructured binding migration with imported dependencies).
